# XRT 库模块裁剪使用指南

## 概述

XRT 库现在支持编译时模块裁剪功能，允许开发者根据实际需求禁用不需要的模块，从而减小最终程序的体积并提高编译速度。

## 裁剪宏定义

### 基础裁剪宏

```c
#define XRT_NO_TIME			// 禁用时间处理
#define XRT_NO_FILE			// 禁用文件操作
#define XRT_NO_THREAD		// 禁用线程管理
#define XRT_NO_COROUTINE	// 禁用协程支持
#define XRT_NO_NETWORK		// 禁用网络基础
#define XRT_NO_CRYPTO		// 禁用加密功能
#define XRT_NO_NETSOCK		// 禁用Socket操作
#define XRT_NO_NETPOLL		// 禁用网络轮询
#define XRT_NO_NETTLS		// 禁用TLS支持
#define XRT_NO_NETLOOP		// 禁用网络事件循环
#define XRT_NO_NETTCP		// 禁用TCP协议
#define XRT_NO_NETUDP		// 禁用UDP协议
#define XRT_NO_XID			// 禁用XID生成
#define XRT_NO_BUFFER		// 禁用缓冲区模块
#define XRT_NO_NETHTTP		// 禁用HTTP协议
#define XRT_NO_ARRAY		// 禁用数组模块
#define XRT_NO_BSMN			// 禁用块结构内存管理
#define XRT_NO_MEMUNIT		// 禁用内存单元
#define XRT_NO_MEMPOOL_FS	// 禁用固定大小内存池
#define XRT_NO_STACK		// 禁用栈模块
#define XRT_NO_AVLTREE		// 禁用AVL树模块
#define XRT_NO_MEMPOOL		// 禁用通用内存池
#define XRT_NO_DICT			// 禁用字典模块
#define XRT_NO_LIST			// 禁用列表模块
#define XRT_NO_VALUE		// 禁用动态类型系统
#define XRT_NO_JNUM			// 禁用数字与字符串转换
#define XRT_NO_JSON			// 禁用JSON处理
#define XRT_NO_TEMPLATE		// 禁用模板引擎
```

### 预定义功能组

```c
#define XRT_MINIMAL         // 最小配置：只保留核心功能
```

## 使用方法

### 1. 基本使用

在包含 `xrt.h` 之前定义裁剪宏：

```c
#define XRT_NO_TEMPLATE
#define XRT_NO_JSON
#define XRT_NO_VALUE
#define XRT_IMPLEMENTATION
#include "xrt.h"
```

### 2. 使用预定义配置组

```c
#define XRT_WEB_SERVER
#define XRT_IMPLEMENTATION
#include "xrt.h"
```

### 3. 条件编译检查

在代码中检查模块是否可用：

```c
#ifndef XRT_NO_TEMPLATE
    // 模板引擎相关代码
    xtemplate tpl = xtplCreate();
    // ...
#endif

#ifndef XRT_NO_JSON
    // JSON处理相关代码
    xrtJsonParseSAX(json_text, len, callback);
#endif
```

## 依赖关系说明

### 强制依赖关系

某些模块之间存在强制依赖关系，系统会自动处理：

- **模板引擎** → 依赖 VALUE系统、DICT、LIST、AVLTree、内存管理层
- **JSON处理** → 依赖 VALUE系统、DICT、LIST、AVLTree、内存管理层  
- **VALUE系统** → 依赖 DICT、LIST、AVLTree、内存管理层
- **DICT/LIST** → 依赖 AVLTree
- **内存池** → 依赖 BSMM、MEMUNIT

### 自动依赖处理

当启用高级模块时，系统会自动启用所需的依赖模块：

```c
// 如果启用模板引擎，系统会自动启用：
// VALUE系统、JNUM、DICT、LIST、AVLTree、BSMM、MEMUNIT、FSMemPool

#define XRT_NO_TEMPLATE  // 禁用模板引擎
// 但如果不禁用VALUE系统，相关依赖仍会被启用
```

## 配置示例

### 示例1：最小化配置

```c
#define XRT_MINIMAL
#define XRT_IMPLEMENTATION
#include "xrt.h"

// 只包含：基础内存管理、字符串处理、操作系统接口、哈希算法
```

### 示例2：自定义配置

```c
// 禁用不需要的高级功能
#define XRT_NO_TEMPLATE
#define XRT_NO_JSON
#define XRT_NO_VALUE
#define XRT_NO_JNUM

// 保留需要的网络功能
// #define XRT_NO_NETWORK    // 不禁用网络
// #define XRT_NO_NETHTTP    // 不禁用HTTP

#define XRT_IMPLEMENTATION
#include "xrt.h"
```

### 示例3：嵌入式系统配置

```c
#define XRT_NO_TIME
#define XRT_NO_FILE
#define XRT_NO_THREAD
#define XRT_NO_COROUTINE
#define XRT_NO_CRYPTO
#define XRT_NO_ARRAY

#define XRT_NO_JSON

#include "xrt.h"
```

## 注意事项

1. **依赖检查**：系统会在编译时自动检查依赖关系冲突
2. **API可用性**：被禁用模块的相关API在编译时会被移除
3. **链接兼容性**：不同裁剪配置生成的目标文件可能不兼容
4. **性能影响**：裁剪掉不需要的模块可以减少程序体积和内存占用
5. **调试便利**：可以逐步启用模块来定位问题

## 故障排除

### 常见错误

1. **依赖冲突错误**：
   ```
   错误: VALUE系统被禁用，但TEMPLATE或JSON模块需要它
   ```
   解决：要么启用VALUE系统，要么同时禁用TEMPLATE和JSON

2. **未定义标识符**：
   ```
   error: 'xtemplate' undeclared
   ```
   解决：检查是否禁用了相关模块但仍在使用其API

3. **链接错误**：
   ```
   undefined reference to `xtplCreate'
   ```
   解决：确保在使用API前检查相应的裁剪宏

## 最佳实践

1. **渐进式裁剪**：从全功能开始，逐步禁用不需要的模块
2. **功能分组**：使用预定义的功能组作为起点
3. **条件编译**：在代码中使用条件编译包装可选功能
4. **文档记录**：记录项目的裁剪配置和原因
5. **回归测试**：每次调整裁剪配置后都要进行完整测试

这样就可以灵活地根据项目需求来裁剪XRT库，既保证了功能完整性，又实现了轻量化的目标。

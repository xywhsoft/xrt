# XRT Builtin TLS Validation and Benchmark Report (2026-03-15)

## Scope

This report records the current dedicated TLS-specific validation for the
finalized XRT foundation work.

It complements the transport compare reports by measuring TLS directly instead
of only as part of the TCP/TLS echo compare matrix.


## Harnesses

### Handshake Throughput and Latency

Source:

- `dev/bench/xnet2/bench_tls_handshake.c`

Build used for this report:

- `gcc dev/bench/xnet2/bench_tls_handshake.c xrt.c -I . -o dev/bench/xnet2/bench_tls_handshake_doc.exe -lws2_32 -liphlpapi`

Run used for this report:

- `dev/bench/xnet2/bench_tls_handshake_doc.exe 32`

What it measures:

- repeated real TLS handshakes over the modern xnet stream path
- handshake latency from connect start to both sides reporting open
- both active handshake rate and whole-loop handshake rate

### Session Resume Throughput and Latency

Source:

- `dev/bench/xnet2/bench_tls_resume.c`

Build used for this report:

- `gcc dev/bench/xnet2/bench_tls_resume.c xrt.c -I . -o dev/bench/xnet2/bench_tls_resume_doc.exe -lws2_32 -liphlpapi`

Run used for this report:

- `dev/bench/xnet2/bench_tls_resume_doc.exe 32`

What it measures:

- one warm full TLS 1.2 handshake to seed mainline session-resume state
- repeated real TLS 1.2 resumed reconnects over the modern xnet stream path
- resumed latency from connect start to both sides reporting open
- both active resumed-handshake rate and whole-loop resumed-handshake rate
- correctness guardrails that reject cold reconnects being reported as resumed


### Boundary and Parser Stress

Source:

- `dev/test_tls_boundary_core.c`

Build used for this report:

- `gcc dev/test_tls_boundary_core.c xrt.c -I . -o dev/test_tls_boundary_core_doc.exe -lws2_32 -liphlpapi`

What it validates:

- fragmented handshake records
- fragmented application-data records
- fragmented close-notify exchange
- socketless TLS drive/feed operation through `xrtTlsDrive` and `xrtTlsFeed`

Read:

- the handshake lane is stable and free of client/server errors
- this is a correctness-oriented dedicated handshake baseline, not yet a final
  optimized production tuning result

### Windows Local Resume Run

Matrix:

- iterations: `32`
- cert: `dev/xnet2_tls_test_cert.pem`
- key: `dev/xnet2_tls_test_key.pem`
- protocol ceiling: `TLS 1.2`

Results:

| Metric | Value |
|---|---:|
| warm handshake ns | `276510100` |
| warm handshake us | `276510.100` |
| resume sum ns | `1102929400` |
| resumed handshakes/s active | `29.014` |
| resume avg us | `34466.544` |
| resume p50 us | `33546.400` |
| resume p95 us | `38730.300` |
| resume p99 us | `38799.600` |
| resume loop elapsed ns | `3368086800` |
| resumed handshakes/s loop | `9.501` |
| resumed client ok count | `32` |
| resumed server ok count | `32` |
| resumed payload ok count | `32` |

Read:

- the dedicated resume lane is now part of the mainline TLS baseline
- the benchmark uses a real exported `xtlsresume` object from a completed warm
  handshake and refuses to count reconnects unless both client and server
  report resumed state
- in this run, resumed handshakes averaged about `34.5 ms` versus a warm
  full-handshake seed cost of about `276.5 ms`, showing an approximately
  `8.0x` lower connect-to-open latency on the measured resume path


### Boundary Stress

Result:

- `TLS boundary handshake/data/close rounds : PASS`

Read:

- the current builtin TLS engine successfully handles fragmented handshake,
  application-data, and close-notify traffic when driven without direct socket
  reads

## Resume Status

The current dedicated resume baseline covers the mainline TLS 1.2
session-resumption path.

There is not yet a separate session-ticket benchmark lane because the current
mainline resume API is session-id based; if a ticket-based mainline path is
added later, it should be reported separately rather than merged into the
numbers above.

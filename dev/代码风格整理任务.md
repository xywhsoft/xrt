## 三、排除规则

以下文件/情况不做风格修改：

1. **明确的第三方库**：仅做换行符统一（CRLF），不改注释、缩进等

   - `lib/jnum.h` — 第三方 JSON 数字解析（有 SPDX 许可证头）
   - `lib/math.h` — 第三方数学库（有来源说明）
   - `lib/crypto.h` — 第三方加密库（5395 行，需确认）
2. **宏定义文件**：仅做换行符统一
   - `lib/memdebug_site_macros_base.h`
   - `lib/memdebug_site_macros_public.h`
   - `lib/memdebug_site_macros_undef.h`
3. **极短文件**（< 20 行）：仅做换行符和缩进检查

## 四、整理规则清单

以下每条规则都有唯一编号，在进度表中用规则编号标注已应用的规则。

### 4.1 缩进与空白

| 编号 | 规则 | 说明 |
|------|------|------|
| R01 | Tab 缩进 | 所有空格缩进转为 Tab（注意：字符串常量内的空格不转换；对齐注释的空格不转换） |
| R02 | CRLF 换行 | 统一为 `\r\n`（仅检查，通过脚本批量处理） |
| R02a | UTF-8 编码 | 所有文件统一使用 UTF-8 编码（无 BOM），发现非 UTF-8 文件需转换 |
| R03 | `#if` 宏缩进 | `#ifdef`/`#if`/`#elif`/`#else`/`#endif` 内部的代码块添加 Tab 缩进 |
| R04 | 括号内外空格 | `if`、`for`、`while`、`switch` 小括号内外都加空格：`if ( condition ) {` |

### 4.2 花括号

| 编号 | 规则 | 说明 |
|------|------|------|
| R05 | 函数定义 `{` 换行 | 函数体 `{` 另起一行 |
| R06 | 语句块 `{` 不换行 | `if`/`for`/`while`/`else` 的 `{` 保持在同一行，前加空格 |
| R07 | 始终使用 `{}` | 单行语句块也必须加 `{}`，不允许省略 |
| R08 | 紧凑单行 | 简单逻辑可写在一行：`if ( x ) { return; }`（已是此风格的不拆分） |

### 4.3 表达式

| 编号 | 规则 | 说明 |
|------|------|------|
| R09 | 冗余括号 | `&&`/`||` 组合条件的每个子表达式都加括号 |
| R10 | 显式类型转换 | 不新增隐式转换，保持现有显式转换风格 |

### 4.4 注释

| 编号 | 规则 | 说明 |
|------|------|------|
| R11 | 函数前注释 | 每个函数前必须有 `//` 单行注释或详细块注释 |
| R12 | 函数内步骤注释 | 功能步骤前加 `//` 注释 |
| R13 | 算法处行内注释 | 位运算、编码转换等处添加说明 |
| R14 | 中文注释 | 注释使用中文（不改动已有的英文注释，新增注释使用中文） |
| R15 | 中文括号注释 | 补充说明使用全角括号加空格：`（ 说明内容 ）` |

### 4.5 空行

| 编号 | 规则 | 说明 |
|------|------|------|
| R16 | 空行分隔 | 调整函数间空行数：耦合函数 0 行、紧密相关 1 行、同模块不同功能集 3 行、不同模块 5+ 行 |

### 4.6 结构体

| 编号 | 规则 | 说明 |
|------|------|------|
| R17 | 结构体注释对齐 | 字段行内注释使用 Tab 对齐到统一列 |

## 五、操作流程

### 5.1 单文件处理流程

```
1. 读取文件全部内容
2. 逐行检查，记录需要修改的位置和对应规则编号
3. 按规则逐一修改（注意：一次只改一种类型，避免冲突）
4. 修改完成后通读一遍，确认无遗漏
5. 更新 spec 进度
6. 如发现 BUG 或疑义，记录到 dev/整理代码问题清单.md
```

### 5.2 修改优先级顺序

为避免冲突，按以下顺序逐项修改：

1. **R02** — 换行符统一为 CRLF（可脚本批量处理）
2. **R02a** — 编码统一为 UTF-8（无 BOM）
3. **R01** — 空格缩进转 Tab
3. **R03** — `#if` 宏内部缩进
4. **R05** — 函数定义 `{` 换行
5. **R07** — 单行语句块补 `{}`（同时应用 R06）
6. **R04** — 括号内外空格
7. **R09** — 冗余括号
8. **R16** — 空行调整
9. **R17** — 结构体注释对齐
10. **R11/R12/R13/R14/R15** — 注释补充与修正

### 5.3 批量处理策略

- **Phase 0**：脚本批量处理 R02（换行符统一）+ R02a（UTF-8 编码统一）
- **Phase 1-N**：逐文件手工处理其余规则

## 七、安全边界

### 7.1 绝对不做的修改

1. **不改变任何变量名、函数名**
2. **不改变任何逻辑判断条件**（只补充括号、空格，不改变求值结果）
3. **不移动函数位置、不合并/拆分函数**
4. **不删除任何代码行**（除非是纯粹的空行调整）
5. **不修改字符串常量内容**
6. **不改变 `#include` 顺序**
7. **不修改宏定义的值**
8. **不修改第三方库的代码风格**（仅换行符统一）

### 7.2 需报告的问题

整理过程中如果发现以下情况，记录到 `dev/整理代码问题清单.md`：

1. 可能存在的 BUG（逻辑错误、内存泄漏风险等）
2. 疑似的死代码或不可达代码
3. 不一致的 API 使用方式
4. 安全隐患（缓冲区溢出风险等）
5. 性能明显可优化的位置

## 九、上下文管理策略

### 9.1 核心原则

**最大化每次会话的工作量，每完成一个文件立即更新 spec 进度。上下文丢失不可怕，只要 spec 是最新的就能无缝恢复。**

### 9.2 极简输出策略

为最大化每次会话处理的文件数量，严格遵守：

1. **不展示代码**：修改后不输出修改前后的代码片段，只输出一句结论（如 `lib/base.h 完成：R01,R03,R07`）
2. **不读不改的不报告**：如果某条规则不需要修改，不输出任何信息
3. **每完成一个文件更新 spec**：用 Edit 工具更新进度表
4. **不重复读 spec**：整个会话只在开头读一次 spec，不在中途重复读取
5. **大文件分段处理**：超过 1000 行的文件分 2-3 段处理，每段 500 行左右
6. **连轴转不休息**：完成一个文件后立即开始下一个，不输出过渡信息

### 9.3 上下文恢复要点

当因上下文溢出或新会话需要恢复任务时，按以下步骤操作：

1. 读取本 spec 文件，找到当前进度
2. 读取 `dev/代码风格规则.md`，重温风格规则
3. 读取 `dev/整理代码问题清单.md`，了解已发现的问题
4. 从最后一个「已完成」的文件继续
5. 如果某个文件标记为「进行中」但未完成，重新从头处理该文件

### 9.4 用户恢复提示词

上下文溢出后，用户在终端发送以下任意一条即可恢复：

- `继续代码风格整理`
- `继续整理`
- `resume style task`

我会自动读取 spec，定位进度，继续执行。

## 十、执行命令

确认 spec 后，按以下顺序开始执行：

```
1. 创建 dev/整理代码问题清单.md（空文件，带标题）
2. 执行 Phase 0（批量换行符统一）
3. 从 Phase 1 开始逐文件处理
```

# 2. C语言 开发习惯

## 2.1 编译与依赖
1. 我常用的编译器为 gcc 和 tcc, 通常使用 tcc 编译器快速验证功能, 再使用 gcc 编译器编译出优化的可执行文件; tcc 也广泛用于执行 C 语言脚本，但应生成尽可能兼容性广泛的代码。
2. Windows 平台下我使用的 gcc 编译器为 mingw64(w64devkit)。
3. gcc 和 tcc 都已加入环境变量, 在 cmd 和 powershell 中都可以直接运行。
4. 我习惯在项目目录添加 build 脚本, windows 系统为 bat 脚本, linux 为 sh 脚本, build 后面不同的内容代表不同的编译类型, 例如 DLL、测试程序等。
5. 我开发的 xrt 项目是一个 C 语言基础运行时, 我大部分 C 语言项目都复用了这个库, 你可以在 D:\git\xrt 或 https://gitee.com/xywhsoft/xrt 找到 xrt 的代码和头文件以了解其功能。
6. 部分项目使用 B/S 架构, 我开发了 xserver 项目让 B/S 架构的程序开发起来更容易, xserver 也缩写为 xs, 它是一个网络服务器项目(HTTP、WebSocket、MQTT、自定义网络协议等), 你可以在 D:\git\xserver 或 https://gitee.com/xywhsoft/xserver 找到 xserver 的代码和头文件以了解其功能。
7. xserver 集成了 xrt(第 5 条的基础运行库)、sqlite、xdo(便捷的数据库操作库) 等功能, 在基于 xserver 开发的项目中遇到类似功能的函数, 应该去 xserver 项目寻找功能实现代码和函数定义等信息。
8. 使用 xserver 开发的项目在根目录能找到 xs 或 xs.exe。
9. 基于 xserver 开发的项目, 直接启动 xs (windows 系统为 xs.exe) 即可进行调试, C 代码为 TCC 运行时编译的, 不需要运行 build.bat 等编译脚本
10. 尽量不要扩展依赖, 如果必须引入新的依赖, 先由我决定是否继续。
11. 引入新的依赖前, 按照衍生依赖越小优先级越高、集成难度越低优先级越高、依赖文件数量越少优先级越高、功能和稳定性越强优先级越好的规则进行排序推荐 (我倾向选择 nothings 即 Sean Barrett 哪种单头文件依赖, 或单个 .c + .h 的依赖)。
12. 在没有其他依赖时, 我通常使用单个 .c 文件进行开发(文件名通常为 main.c, 有些会以项目名称命名), 我会将所有依赖统一在这个 .c 文件中引用, 然后按照功能分类创建不同的目录和多个 .h 文件来实现各个功能模块。
13. 由于只有单个 .c 文件, 编译会非常简单, 所以我不使用 make 工具, 而是直接使用 gcc、tcc 命令行编译程序, 并实现编译脚本简化这一过程 (参考第 4 条)。
14. 由于我已经在 .c 文件中引用所有依赖了, 因此在 .h 文件中, 通常不需要额外引用其他依赖的头文件, 除非这个头文件只在某个特定模块中使用。

## 2.2 代码规范

1. 使用匈牙利命名法命名变量, 使用小写数据类型做前缀, 数据类型表：整数为 i, 浮点数为 f, 字符串为 s, 时间为 t, 布尔值为 b, 句柄为 h, 指针为 p, 函数为 proc, 数组为 arr, 字典为 tbl, 列表为 lst。
2. 定义函数时, 花括号单起一行, 例如：
	```
	void FunctionName() 
	{
		// 函数体花括号单起一行
	}
	```
3. if、else、for、while 等语句的花括号则与语句保持在同一行, 小括号两侧内外都添加空格, 例如：
	```
	if ( condition ) {
		// code
	} else {
		// code
	}

	for ( int i = 0; i < 10; i++ ) {
		// code
	}

	while ( condition ) {
		// code
	}
	```
4. 条件判断语句中如果有 &&、|| 等组合条件, 我习惯为所有组合条件添加括号, 以避免特殊情况下运算符优先级产生问题, 例如:
	```
	if ( (a == 1) && ((b == 2) || (c == 3)) ) {
		// code
	}
	```
5. 我习惯使用空行来区分功能块, 高度相关的功能我会留 1 个空行来表示两个函数之间联系紧密；如果两个函数是完全耦合的关系(例如回调函数)我不会留空行；一个模块之间的函数, 我会留 3 个空行；如果同一个文件中, 存在多个功能迥异的模块, 我会根据他们是否需要被严格区分, 决定留 5 个或 11 个空行(11个空行常见于代码行数多的文件, 这样做是为了让我在快速浏览代码时能更容易发现这些空行)
